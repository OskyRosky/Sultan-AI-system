# 02 Strategy Inputs Contract

## Purpose

This document defines the governed input contract for 05 Strategy Engine.

The contract exists because no future signal, rule, or strategy candidate may be legitimate unless it can trace back to admissible research evidence, findings, or hypotheses from 04 Research Layer.

This block is contractual and auditable. It does not consume real data and does not assume that real approved findings or hypotheses exist.

## Source Boundary

05 Strategy Engine may only consume governed artifacts from 04 Research Layer after real research execution, human review, quality review, limitations, and caveats have been documented.

04 Research Layer is currently closed as a framework, not as empirical research closure. Therefore, this contract defines admissibility for future inputs. It does not mark any current real input as eligible.

## Accepted Conceptual Input Types

### Research Evidence Input

Research Evidence Input represents evidence generated by 04. Evidence may support research interpretation, but isolated evidence never creates a signal or strategy candidate.

Required conceptual fields:

- `evidence_id`
- `source_layer`
- `source_component`
- `evidence_type`
- `feature_reference`
- `target_reference`, when applicable
- `regime_reference`, when applicable
- `temporal_scope`
- `asset_scope`
- `timeframe_scope`
- `evidence_status`
- `approval_status`
- `created_at`
- `limitations`
- `audit_reference`

### Finding Input

Finding Input represents a documented research finding from 04.

Required conceptual fields:

- `finding_id`
- `linked_evidence_ids`
- `finding_summary`
- `finding_type`
- `regime_scope`, when applicable
- `stability_assessment`
- `informativeness_assessment`
- `falsification_reference` or documented limitations
- `finding_status`
- `approval_status`
- `closure_reference`
- `audit_reference`

### Hypothesis Input

Hypothesis Input represents a research hypothesis authorized for possible future strategy design.

Required conceptual fields:

- `hypothesis_id`
- `linked_finding_ids`
- `linked_evidence_ids`
- `hypothesis_statement`
- `expected_behavior`
- `applicable_regime_context`, when applicable
- `falsification_criteria`
- `limitations`
- `approval_status`
- `eligible_for_strategy_design`
- `audit_reference`

### Strategy Input Eligibility Decision

Strategy Input Eligibility Decision is the explicit 05 decision that records whether an input may feed later conceptual signal and rule design.

Required conceptual fields:

- `input_id`
- `input_type`
- `eligibility_status`
- `decision_reason`
- `required_approvals_present`
- `falsification_criteria_present`, when applicable
- `traceability_complete`
- `limitations_acknowledged`
- `decided_at`
- `audit_reference`

## Status Model

The contract separates documentation state from strategy design eligibility.

Approval states:

- `draft`
- `pending_review`
- `approved`
- `rejected`
- `closed`

Eligibility states:

- `ineligible_for_strategy_design`
- `eligible_for_strategy_design`

An input can be documented and still be ineligible. An input can be approved as research context and still imply no edge, no profitability, no signal, and no trading permission.

## Eligibility Requirements

At minimum, an eligible input must satisfy:

1. It originates from 04 Research Layer.
2. It has a stable audit reference.
3. It has complete traceability to evidence, findings, or hypotheses as applicable.
4. It has approval status `approved`.
5. It declares limitations.
6. It preserves temporal, asset, timeframe, and regime context when applicable.
7. If it is a hypothesis, it has falsification criteria.
8. If it is a finding, it is approved and linked to evidence.
9. If it is evidence, it cannot alone create strategy design eligibility.

## Explicit Prohibitions

This contract must not be used to:

- Create real signals.
- Create real rules.
- Create real strategy candidates.
- Run backtests.
- Calculate PnL.
- Calculate Sharpe, Sortino, Calmar, or return metrics.
- Optimize parameters.
- Read PostgreSQL.
- Read Parquet.
- Load datasets.
- Confirm edge.
- Claim profitable behavior.
- Execute orders.

## Non-Edge Disclaimer

Eligibility for strategy design means only that the input may be considered by future 05 conceptual design layers. It does not imply confirmed edge, profitable performance, robustness, causality, deployability, execution readiness, or approval for trading.

## Implementation Scope

Block 2 implements a small in-memory contract representation and synthetic tests. It does not integrate technically with 04 outputs and does not read any real research artifact.
